# Available scripts

This document describes the scripts in `bin/`.

> **Type:** reference · **Audience:** operator · **Last reviewed:** 2026-06-11

## Overview

| Script | Primary use |
| --- | --- |
| `bin/check-env.sh` | Validate the keys in `.env` against `.env.example`. |
| `bin/deploy.sh` | Sync the repository, then build, deploy, or roll back the NixOS configuration. |
| `bin/apply.sh` | Update the revision of an existing app, commit, push, and deploy. |
| `bin/app-create.sh` | Add an app from a Nix proposal generated by the API. |
| `bin/app-rollback.sh` | Revert an app to a specific revision. |
| `bin/backup.sh` | Run restic backup operations (run, verify, restore, snapshots). |
| `bin/hl-deploy-run` | Wrapper invoked by `systemd-run` to launch a detached deployment job. |
| `bin/hl-backup-run` | Wrapper invoked by `systemd-run` to launch a detached backup job. |
| `bin/docs-check.sh` | Verify relative markdown links in `README.md` and `docs/` resolve. |

## `bin/check-env.sh`

Compares an example file against a target file.

```bash
bash bin/check-env.sh .env.example .env
```

Parameters:

| Position | Default | Description |
| --- | --- | --- |
| `1` | `.env.example` | Reference file. |
| `2` | `.env` | File to check. |

Behavior:

- fails if a key in the target file does not exist in the example;
- reports keys missing from the target file;
- accepts commented lines, blank lines, and `export` prefixes.

## `bin/deploy.sh`

The central deployment script.

```bash
sudo bash bin/deploy.sh /home/admin/homelab switch
```

Parameters:

| Position | Default | Description |
| --- | --- | --- |
| `1` | `/home/admin/homelab` | Repository directory on the host. |
| `2` | `switch` | Mode: `dry-run`, `build`, `switch`, `rollback`. |
| `3` | empty | Optional target, notably a rollback generation or an app target. |

Variables used:

| Variable | Description |
| --- | --- |
| `HOMELAB_DEPLOY_REF` | Git reference to deploy; defaults to `origin/main`. |
| `HOMELAB_FLAKE_REF` | Flake reference; defaults to `path:<dir>#homelab`. |
| `HOMELAB_REPO_USER` | User that owns the checkout. |
| `HOMELAB_REPO_URL` | Preferred Git URL. |
| `REPO_URL` | Fallback Git URL read from `.env`. |
| `HOMELAB_GIT_TOKEN_FILE` | Git token; defaults to `/run/secrets/git_token`. |
| `SUDO` | sudo binary; defaults to `/run/wrappers/bin/sudo`. |

Modes:

```bash
sudo bash bin/deploy.sh /home/admin/homelab dry-run
sudo bash bin/deploy.sh /home/admin/homelab build
sudo bash bin/deploy.sh /home/admin/homelab switch
sudo bash bin/deploy.sh /home/admin/homelab rollback
sudo bash bin/deploy.sh /home/admin/homelab rollback 42
```

In `switch` mode, the script arms an automatic rollback, applies the configuration, then disarms the rollback if a default route exists and `sshd` is active.

## `bin/apply.sh`

Updates a declarative app that has both `repo` and `rev`.

```bash
sudo bash bin/apply.sh whoami /home/admin/homelab
```

Parameters:

| Position | Description |
| --- | --- |
| `1` | App name, matching `apps/<app>.nix`. |
| `2` | Repository directory; defaults to `/home/admin/homelab`. |

Behavior:

1. reads `repo` and `rev` from `apps/<app>.nix`;
2. resolves `HEAD` via `git ls-remote`;
3. replaces `rev` if a newer revision exists;
4. commits with `apply: bump <app> <old> -> <latest>`;
5. pushes to `main`;
6. runs `bin/deploy.sh`.

The script reads the Git token from `/run/secrets/git_token`.

## `bin/app-create.sh`

Creates an app from a proposal file.

```bash
sudo bash bin/app-create.sh demo /var/lib/homelab/app-create-demo.nix /home/admin/homelab none
```

Parameters:

| Position | Description |
| --- | --- |
| `1` | App name. |
| `2` | Path to the Nix proposal file. |
| `3` | Repository directory; defaults to `/home/admin/homelab`. |
| `4` | Deploy mode: `none`, `dry-run`, `build`, `switch`. |

Modes:

| Mode | Behavior |
| --- | --- |
| `none` | Commit on an `app-create/...` branch, then push that branch. |
| `dry-run` | Commit on a branch, push, then `bin/deploy.sh dry-run`. |
| `build` | Commit on a branch, push, then `bin/deploy.sh build`. |
| `switch` | Commit on `main`, push to `main`, deployment by CI. |

## `bin/app-rollback.sh`

Repositions an app onto a specific revision.

```bash
sudo bash bin/app-rollback.sh demo 0000000000000000000000000000000000000000 /home/admin/homelab
```

Parameters:

| Position | Description |
| --- | --- |
| `1` | App name. |
| `2` | Target revision; hexadecimal commit SHA, 7–40 chars. |
| `3` | Repository directory; defaults to `/home/admin/homelab`. |

Behavior:

1. validates the app name and the target revision;
2. replaces `rev` in `apps/<app>.nix`;
3. commits with `rollback: <app> <old> -> <rev>`;
4. pushes to `main`;
5. runs `bin/deploy.sh switch app:<app>:<rev>`.

## `bin/backup.sh`

Drives restic backup operations on the host. It backs the runtime actions exposed under `/v1/backups/*`.

Supported actions: `run`, `verify`, `restore-test`, `restore`, and `snapshots`.

## `bin/hl-deploy-run` and `bin/hl-backup-run`

These small wrappers are launched by `systemd-run` as detached units so that long-running jobs survive the SSH session and any `tailscaled` restart.

- `bin/hl-deploy-run` runs a deployment job (used by the `control-api` deployment flow).
- `bin/hl-backup-run` runs a backup job (used by the `control-api` backup flow).

## Best practices

- Run these scripts on the target NixOS host, not from a Windows workstation.
- Never print the contents of `/run/secrets/git_token`.
- Read the systemd logs after long-running operations.
- Test with `dry-run` or `build` before `switch` when a change is risky.
